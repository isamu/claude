# Claude Code 運用ガイド

## 心構え

コードの品質管理は、人間のチーム開発と同じ。AI は優秀な仲間。きれいなコードを書ける環境を整える PM になろう。

CLAUDE.md と skill をメンテナンスして、AI がルールの中でのびのび書けるようにするのが大切。AI はコードの量を増やしがちなので、人間がメンテできるきれいなコードを書かせることを意識する。人間が読めなくなったら管理できない。

## 品質レベルに応じた使い分け

### 共通（全プロジェクト必須）

- lint / unit test / formatter は必須
- CI でもしっかりテストを回す

### プロダクションコード（きちんと管理するソフトウェア）

- 指示は粒度を細かくし、出力に対してコードレビューを行う
  - DRY でないコード、大きな関数、CLAUDE.md のルール無視が起きやすいため、レビューは重要
- AI レビュー等の GitHub ツール（CodeRabbit など）を導入し、そのフィードバックを Claude に読み込ませて対応させる
- PR はレビュー後にマージ

### プロトタイプ・使い捨てツール

- 粒度の粗い指示でも OK
- 動作確認 + 簡単な PR で十分

## 情報の永続化

チャットは消える。必要な情報は必ず永続的な場所に残す。

- 課題 → GitHub issue を立てる
- 計画 → リポジトリの `plans/` にファイルで残す
- 実装の経緯・判断 → commit メッセージや PR コメントに書く
- 重要な議論 → issue / PR コメントに転記する

## ドキュメント管理

- Markdown + GitHub が最適解
- 専用ツールは長期的に存続するかわからない（Evernote の末路）ので、テキスト + 自前管理が一番

## 活用範囲

プログラミングに限らず、ドキュメント作成やツールの定期実行にも活用できる。LLM を AI OS として捉え、あらゆるタスクの実行レイヤーとして使う。

Claude Code はターミナルで CLI を使えるので、MCP がなくても CLI で提供されているものなら何でも使える。curl や fetch で Web API も叩けるので、Web API が公開されているサービスなら何でも skill や対話で連携できる。これを活かそう。

## 自動化の方針

### Skill で定型化

- 定型タスクやリポジトリ固有のワークフローは skill として作成する

### プログラム化 vs Skill

- プログラム化できるもの → コードとして書き、unit test / generator を作る
- プログラム化が難しいもの → skill で code + LLM を組み合わせて対応する
